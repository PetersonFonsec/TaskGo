# ADR-004: Explicit Provider Commands With Transactional Audit

## Status

Accepted

## Date

2026-07-02

## Context

Provider approval, rejection, blocking, and unblocking have distinct authorization, validation, justification, and state-transition rules. The PRD requires every sensitive administrative action to have a durable audit record that cannot diverge from the resulting provider state.

## Decision

Expose one command endpoint for each provider transition under `/admin/providers/:id`. Execute the validated transition, provider lifecycle event, and immutable audit entry in one Prisma transaction. A failed audit write rolls back the state change. Read endpoints remain separate, paginated queries.

## Alternatives Considered

### Alternative 1: Generic Status Update Endpoint

- **Description**: Accept any target provider status through one patch endpoint.
- **Pros**: Small API surface and less controller code.
- **Cons**: Centralizes branching, weakens intent, and makes per-command authorization and validation easier to bypass.
- **Why rejected**: Explicit commands make security rules and audit semantics testable.

### Alternative 2: Asynchronous Audit Events

- **Description**: Commit the state change and write audit records from an internal event handler.
- **Pros**: Lower command latency and loose coupling.
- **Cons**: Delivery failure can produce unaudited state changes.
- **Why rejected**: The PRD requires complete audit coverage over throughput optimization.

## Consequences

### Positive

- State and audit records remain consistent.
- Each transition has a focused contract and authorization rule.
- Retried commands can define deterministic conflict behavior.

### Negative

- More endpoints and handlers must be maintained.
- Audit persistence contributes to write latency.

### Risks

- Concurrent commands could race; use conditional updates against the expected current status and return conflict on mismatch.
- Audit payloads could leak unnecessary personal data; store identifiers and state deltas, not full provider records.

## Implementation Notes

Model provider status independently from the existing `verified` boolean and migrate existing providers deterministically. Index provider status queues and audit filters used by the PRD.
