# ADR-002: Separate Backoffice Frontend With Shared API

## Status

Accepted

## Date

2026-07-02

## Context

The current Nx workspace contains one Angular application for customers and providers and one NestJS API containing all marketplace domains. The Backoffice requires a distinct administrative surface while still operating on the same provider, service, order, and payment records.

## Decision

Add a separate Angular application for the Backoffice inside the existing Nx workspace. Keep the existing NestJS application and expose administrative capabilities under the `/admin` API namespace. Both applications use the existing PostgreSQL database through the NestJS API; the Backoffice never accesses storage directly.

## Alternatives Considered

### Alternative 1: Add Administrative Routes to the Existing Frontend

- **Description**: Host Backoffice screens under `/admin` in the current Angular application.
- **Pros**: Lowest setup cost and maximum reuse of frontend infrastructure.
- **Cons**: Couples public and privileged surfaces and increases the risk of accidental route or bundle exposure.
- **Why rejected**: The user selected a separately deployable administrative frontend boundary.

### Alternative 2: Separate Frontend and Separate API

- **Description**: Build dedicated Angular and NestJS applications for administration.
- **Pros**: Strongest runtime and deployment isolation.
- **Cons**: Duplicates domain access, authentication infrastructure, observability, and operational maintenance.
- **Why rejected**: A second API adds complexity without a current data or scaling requirement.

## Consequences

### Positive

- Public and administrative navigation, assets, and deployments remain isolated.
- The API preserves one source of truth for marketplace rules and data.
- The Nx workspace can share tooling without coupling application routes.

### Negative

- The workspace gains another build and deployment target.
- Shared UI primitives require explicit library extraction or controlled duplication.

### Risks

- Shared API controllers could blur boundaries; mitigate with a dedicated `AdminModule`, `/admin` namespace, and administrative guard.
- Frontend duplication may grow; extract only stable primitives that both applications actually need.

## Implementation Notes

Generate an Nx Angular application named `backoffice`. Add a NestJS `AdminModule` that composes provider operations without exposing administrative commands through public controllers.
