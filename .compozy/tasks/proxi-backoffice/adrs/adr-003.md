# ADR-003: Dedicated Administrative Identities With Shared JWT Secret

## Status

Accepted

## Date

2026-07-02

## Context

Proxi customer and provider identities live in `User`, and current JWTs contain only the user identifier. Backoffice operators require an independent lifecycle, four fixed roles, immediate deactivation, and tokens that public authentication cannot be mistaken for administrative authorization.

## Decision

Store operators in a dedicated `AdminUser` model with email, password hash, fixed role, active status, and lifecycle timestamps. Authenticate through `/admin/auth/login`. Administrative JWTs use the existing signing secret but include an explicit administrative token kind, administrator identifier, role, and token version. `AdminAuthGuard` rejects ordinary user tokens, loads the current administrator on every request, and verifies active status and token version before role authorization.

## Alternatives Considered

### Alternative 1: Extend the Existing User Model

- **Description**: Add nullable administrative role and active fields to `User`.
- **Pros**: Reuses all identity and authentication code.
- **Cons**: Couples internal operators to marketplace identities and complicates lifecycle constraints.
- **Why rejected**: Administrative identities must remain independent from customers and providers.

### Alternative 2: Use a Separate JWT Secret

- **Description**: Sign administrative tokens with dedicated credentials and audience.
- **Pros**: Stronger cryptographic isolation and simpler token discrimination.
- **Cons**: Adds secret rotation and environment management.
- **Why rejected**: The selected design reuses the current secret and enforces logical separation through claims and guards.

## Consequences

### Positive

- Operator deactivation and role changes do not affect marketplace users.
- Token kind and database validation prevent ordinary JWTs from reaching administrative endpoints.
- Token versioning supports immediate invalidation after security changes.

### Negative

- Sharing a signing secret increases blast radius if that secret is compromised.
- Every administrative request requires an operator lookup unless safely cached.

### Risks

- A missing token-kind check could allow token confusion; require negative E2E tests for every administrative route family.
- Role changes could leave old claims usable; compare token version and current role against storage on each request.

## Implementation Notes

Keep password hashing consistent with the existing bcrypt dependency. Never accept an administrative role from an unverified JWT without matching the active `AdminUser` record.
