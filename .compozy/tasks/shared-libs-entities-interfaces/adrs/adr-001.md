# ADR-001: Prioritize Shared Domain Language for Authentication and Profile

## Status

Accepted

## Date

2026-07-09

## Context

TaskGo has two Angular frontends and one NestJS backend in a monorepo. The workspace already contains a shared library area, but authentication and profile concepts are still represented through separate backend DTOs, backend interfaces, frontend models, backoffice session models, enums, and local aliases. This creates a risk that user, role, session, address, and profile concepts drift across applications.

The product goal is to avoid divergence and standardize the domain language while proving value in a focused MVP. The selected first flow is authentication and user profile because it affects all applications and gives the team a clear way to evaluate whether shared contracts improve consistency.

## Decision

Prioritize a domain language-first MVP for authentication and profile. The MVP will define shared public concepts for the user/profile domain and make frontend, backoffice, and backend flows use the same names and meanings for those concepts. Request and response contracts remain part of the value, but the primary product outcome is a consistent domain vocabulary that prevents duplicated or conflicting interpretations.

## Alternatives Considered

### Alternative 1: Shared contracts-first

- **Description**: Focus first on public request and response contracts for authentication and profile.
- **Pros**: Fastest route to remove duplicated API-facing models and reduce frontend/backend mismatch.
- **Cons**: May leave deeper domain concepts, enums, and value objects fragmented across apps.
- **Why rejected**: The user selected a broader domain-language outcome over a contract-only MVP.

### Alternative 2: Broad shared foundation

- **Description**: Create a broad shared foundation for contracts, entities, enums, value objects, and reusable organization rules across the monorepo.
- **Pros**: Establishes a comprehensive foundation for future sharing.
- **Cons**: Higher scope and adoption risk for the first iteration.
- **Why rejected**: Too broad for proving MVP value in one critical flow.

## Consequences

### Positive

- Authentication and profile concepts gain a single product vocabulary across the three applications.
- The MVP can demonstrate value without attempting to migrate every domain at once.
- Future shared contracts can build on clearer domain boundaries.

### Negative

- Some request/response duplication may remain until later phases.
- The team must decide which concepts are public domain language and which are app-specific presentation details.

### Risks

- Shared domain concepts could become too broad if the MVP absorbs unrelated profile-adjacent behavior.
- Backend implementation details could leak into frontend expectations unless public concepts are clearly separated from internal concepts.

## Implementation Notes

Implementation details will be defined in the TechSpec. The PRD should remain focused on user-facing and team-facing outcomes.
