# ADR-002: Use Pure Serializable TypeScript Contracts for Auth and Profile

## Status

Accepted

## Date

2026-07-09

## Context

The TaskGo monorepo contains two Angular applications and one NestJS backend. Authentication and profile types currently appear as separate frontend models, backoffice session models, backend DTO classes, Prisma-backed enums, and backend entity/value-object classes. The shared library area already exists, but it does not yet expose a stable public contract surface for authentication and profile flows.

The selected PRD approach is domain language-first, with a focused MVP for authentication and profile. Codebase exploration showed that backend DTOs contain decorators and runtime validation concerns, while the frontends need JSON-serializable shapes. It also showed sensitive or backend-only fields in some local public models, including password/passwordHash and broad relational data.

## Decision

Create pure TypeScript, JSON-serializable shared contracts in `libs/shared` for the MVP authentication and profile surface. The MVP covers login/session/me, public profile, registration payloads, and profile update payloads. Backend NestJS DTO classes will remain as framework-specific adapters that implement or map to the shared contract types instead of becoming the shared source themselves.

Replace the selected local auth/profile models directly during the MVP rather than keeping parallel compatibility models. Validate the change with build/type-check coverage for backend, frontend, and backoffice.

## Alternatives Considered

### Alternative 1: Shared classes with validation and behavior

- **Description**: Share classes with validation logic and methods across backend and frontends.
- **Pros**: Centralizes validation behavior and can reduce backend wrapper code.
- **Cons**: Couples Angular applications to backend validation/decorator concerns and risks shipping runtime behavior where only type contracts are needed.
- **Why rejected**: The MVP needs stable public contracts, not shared framework behavior.

### Alternative 2: Generated contracts from backend

- **Description**: Generate frontend contract types from backend API metadata.
- **Pros**: Keeps contracts close to backend endpoints and can reduce manual drift.
- **Cons**: Adds a generation workflow and extra moving parts before the shared vocabulary is stabilized.
- **Why rejected**: Too heavy for the MVP and less aligned with the domain language-first decision.

### Alternative 3: Incremental compatibility aliases

- **Description**: Keep local models as temporary aliases or extensions of shared contracts and migrate gradually.
- **Pros**: Lowers migration risk and lets each app move independently.
- **Cons**: Keeps duplicate names alive during the MVP and weakens the success metric of removing duplicated public auth/profile models.
- **Why rejected**: The selected migration strategy is a direct MVP replacement for the chosen scope.

## Consequences

### Positive

- Frontend, backoffice, and backend consume the same public auth/profile vocabulary.
- Shared contracts stay independent from NestJS decorators, Prisma runtime types, and Angular UI concerns.
- The MVP can remove duplicated public models rather than introducing a long compatibility layer.

### Negative

- Backend DTOs still need adapter classes or mappings for validation.
- Direct replacement requires careful coordination across the three projects in one implementation slice.
- Build/type-check validation may catch shape mismatches that require endpoint response cleanup.

### Risks

- Shared contracts may accidentally include backend-only fields if public boundaries are not enforced during implementation.
- Existing app code may rely on overly broad local user shapes, requiring small local view-models for presentation-only data.

## Implementation Notes

- Prefer `type` and `interface` exports with primitive/JSON-safe field types.
- Treat `bigint`, `Date`, Prisma enums, password fields, password hashes, and broad relational data as internal unless explicitly serialized into public contracts.
- Keep endpoint paths and persistence unchanged unless a contract mismatch forces a small response mapping adjustment.
- Validation evidence should include build/type-check of backend, frontend, and backoffice.
