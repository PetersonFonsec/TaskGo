# ADR-001: Provider Availability Slots as Backend Source of Truth

## Status

Accepted

## Date

2026-06-20

## Context

Customers need to view a provider profile, inspect available dates and times, and request an appointment from the `customer/:id` route. The existing frontend page already loads provider details through `SingleUser`, and the existing backend order flow already stores `Order.scheduledFor` and accepts `scheduledFor` in `CreateOrderDto`.

The missing technical boundary is availability. `Service.availability` exists as JSON, but the frontend should not be responsible for interpreting availability rules, filtering blocked times, or checking existing orders. Without a backend availability contract, two customers could see the same open time and create conflicting appointment requests.

## Decision

Implement provider availability as a backend-owned contract. The backend will expose an endpoint that returns available appointment slots for a provider and date range. The service will derive candidate slots from active provider services and their `availability` JSON, then remove slots already occupied by pending or confirmed orders.

The customer profile page will call this endpoint, render the calendar and time selector, and submit the selected ISO datetime through the existing `POST /order` flow using `scheduledFor`.

## Alternatives Considered

### Alternative 1: Frontend-Derived Slots From Provider Payload

- **Description**: Reuse `provider.services[].availability` in Angular, calculate calendar states locally, and submit `scheduledFor` directly.
- **Pros**: Fastest implementation and no new backend endpoint.
- **Cons**: Duplicates business rules in the browser, cannot reliably exclude conflicting orders, and creates inconsistent behavior if multiple clients book the same slot.
- **Why rejected**: Availability is a scheduling rule and must be authoritative on the server.

### Alternative 2: Mocked Schedule UI First

- **Description**: Build the booking screen with static dates and times, then integrate real availability later.
- **Pros**: Provides quick visual progress and can match the provided mockup closely.
- **Cons**: Delays the real workflow, risks throwaway UI state, and does not validate the customer booking journey end to end.
- **Why rejected**: The feature value depends on real appointment selection, not only visual parity.

## Consequences

### Positive

- Keeps availability and conflict rules centralized in the backend.
- Lets the frontend render a deterministic UI contract instead of parsing arbitrary provider service JSON.
- Reuses the existing order creation flow and `scheduledFor` storage.
- Creates a clean extension point for future provider schedule management.

### Negative

- Requires backend work before the final frontend booking flow is complete.
- Requires a normalized interpretation of the current `Service.availability` JSON.
- Adds tests around date/time boundaries and order status filtering.

### Risks

- Existing services may not have complete or consistent `availability` data. Mitigation: return an empty slot list with clear frontend empty states and document the expected JSON shape.
- Timezone handling can produce off-by-one-day errors. Mitigation: accept date-range query params, return ISO datetimes, and document that slots are interpreted in the application's configured timezone for MVP.
- The order creation endpoint may still accept conflicting `scheduledFor` values. Mitigation: add server-side validation in order creation before persisting scheduled orders.

## Implementation Notes

- Add a provider availability endpoint, for example `GET /provider/:id/availability?from=YYYY-MM-DD&to=YYYY-MM-DD&serviceId=...`.
- Add a backend DTO and response model with available dates, slots, duration, service id, and pricing summary.
- Reuse `Order.scheduledFor` and active order statuses to block unavailable slots.
- Update Angular `Provider` service to fetch availability and update `SingleUser` to render date, time, and appointment summary.
