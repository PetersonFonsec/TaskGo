# ADR-001: Hybrid Search Map Experience

## Status

Accepted

## Date

2026-06-14

## Context

Proxi is a services marketplace where customers compare nearby providers. The search experience already depends on provider location data, ratings, pricing, and trust signals. A map can improve spatial understanding, but a map-first experience could reduce the visibility of comparison signals that matter for service selection.

The product needs a map experience that works well on mobile and desktop while keeping the provider list useful as the main comparison surface.

## Decision

Proxi will use a hybrid search map experience. The provider list remains the primary search interface, and the map acts as a visual aid for proximity and location context.

On desktop, the search experience should support a split layout with the list and map visible together. On mobile, the map should appear in a compact or expandable form so it does not displace the provider list. Provider popups should let users open a provider profile through the parent search experience.

## Alternatives Considered

### Alternative 1: Map-Led Discovery

- **Description**: Make the map the primary search surface, with the list acting as supporting content.
- **Pros**: Maximizes location visibility and encourages spatial exploration.
- **Cons**: Makes ratings, prices, trust badges, and service comparison less scannable.
- **Why rejected**: Proxi users need to compare service quality and trust signals, not only distance.

### Alternative 2: Reusable Map Block

- **Description**: Build a generic map block with minimal product behavior.
- **Pros**: Simple, reusable, and lower product scope.
- **Cons**: Provides limited value to the provider discovery journey.
- **Why rejected**: The initial value should improve search decisions, not only render coordinates.

## Consequences

### Positive

- Preserves the provider list as the main decision surface.
- Adds location context without overwhelming mobile users.
- Supports desktop users who benefit from seeing list and map together.
- Keeps provider profile navigation owned by the parent search flow.

### Negative

- Requires responsive behavior rules for mobile and desktop.
- Requires careful empty-state handling so the compact map does not feel broken.

### Risks

- Users may ignore the map if it is too compact on mobile. Mitigation: make the compact state discoverable and useful.
- Users may overvalue distance over trust signals. Mitigation: show rating, pricing, verification, and premium cues in provider popups.

## Implementation Notes

Implementation details should be defined in the TechSpec. The PRD should preserve the product boundary: the map supports search, shows nearby providers, and emits profile intent to the parent experience.
