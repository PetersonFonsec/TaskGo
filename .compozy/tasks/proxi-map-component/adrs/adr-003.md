# ADR-003: Local Map Provider Contract and Search Page Adaptation

## Status

Accepted

## Date

2026-06-14

## Context

The current customer search page receives provider data from the provider service using an endpoint-specific response shape. The map requires a stable, small contract with id, display details, latitude, longitude, distance, and trust flags. Reusing the backend response shape directly would couple the map to endpoint internals and make future provider response changes more expensive.

## Decision

Define local exported interfaces in `ProxiMapComponent` for `ProxiMapProvider` and `ProxiMapLocation`. The customer search page will adapt provider API data into this map contract before passing it to the component.

The search page will handle navigation to `/customer/:userId` after receiving the component's `viewProfile` event.

## Alternatives Considered

### Alternative 1: Reuse Provider Endpoint Shape Directly

- **Description**: Pass the raw provider response into the map component.
- **Pros**: Less mapping code in the first implementation.
- **Cons**: Couples the UI component to a mutable API response shape.
- **Why rejected**: The map should have a stable UI-facing contract independent of API details.

### Alternative 2: Create Global Shared Map Model

- **Description**: Add map-specific models to a shared service or global models directory.
- **Pros**: Centralizes map data types.
- **Cons**: Adds broader abstraction before multiple consumers exist.
- **Why rejected**: YAGNI favors keeping the contract local until reuse requires promotion.

## Consequences

### Positive

- Keeps `ProxiMapComponent` simple, typed, and reusable.
- Keeps backend response changes isolated in the search page adapter.
- Avoids premature global model abstractions.

### Negative

- The search page must maintain a mapping function.
- Other future map consumers may initially duplicate mapping logic until a shared adapter becomes justified.

### Risks

- Raw provider data may not always include complete location or display fields. Mitigation: filter invalid coordinates and provide safe display fallbacks in the adapter.

## Implementation Notes

The local contract should use the field names requested by the feature: `id`, `name`, `service`, `rating`, `priceFrom`, `lat`, `lng`, `distanceKm`, `photoUrl`, `premium`, and `verified`.
