# ADR-003: Verify Adaptive Navigation with Cypress End-to-End Tests

## Status

Accepted

## Date

2026-07-23

## Context

The change affects responsive rendering, route hierarchy, authorization guards, keyboard focus, navigation state, and drawer timing. The repository supports Jasmine/Karma component tests and Cypress browser tests. The selected delivery scope requires browser-level verification only.

## Decision

Use Cypress as the mandatory verification layer for this feature. Cover authenticated navigation at mobile and desktop viewports, role-filtered destinations, current-page feedback, every dismissal path, keyboard focus behavior, route guards, corrected dynamic user paths, direct entry to order routes, and the 200 ms drawer transition requirement.

Do not add feature-specific Jasmine unit or Angular integration tests in this delivery.

## Alternatives Considered

### Alternative 1: Unit, Router Integration, and Cypress Tests

- **Description**: Combine focused component and configuration tests with browser-level flows.
- **Pros**: Provides faster diagnosis, isolated edge-case coverage, and complete user-flow confidence.
- **Cons**: Increases implementation and maintenance scope.
- **Why rejected**: The chosen scope prioritizes browser-level validation only.

### Alternative 2: Unit and Router Integration Tests Only

- **Description**: Test navigation logic, components, route composition, and guards without a real-browser flow.
- **Pros**: Runs quickly and diagnoses failures precisely.
- **Cons**: Does not validate responsive layout, real focus behavior, or the complete interaction.
- **Why rejected**: The primary risk is the cross-viewport user journey.

## Consequences

### Positive

- Verification exercises the behavior users experience in a real browser.
- One test layer covers responsive rendering, routing, guards, and keyboard interaction.
- Existing Cypress infrastructure and selectors can be extended.

### Negative

- Failures may be slower and harder to diagnose than focused unit failures.
- Edge-case coverage of pure configuration logic is more expensive.
- The suite may take longer and be more sensitive to environment timing.

### Risks

- Timing assertions may become flaky; use browser timing measurement and Cypress retryability without fixed waits.
- Browser-only coverage may miss isolated configuration branches; exercise representative users for every supported role.
- Route fixtures may hide authorization regressions; include direct URL entry and denied-route cases.

## Implementation Notes

Create a focused non-skipped Cypress specification for adaptive navigation. Use stable accessible selectors. Run mobile tests below 768 px and desktop tests at or above 768 px. Measure the open transition from activation until the drawer reaches its open state and require at most 200 ms.
