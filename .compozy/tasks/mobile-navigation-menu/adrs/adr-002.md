# ADR-002: Centralize Authenticated Navigation in an Adaptive Shell

## Status

Accepted

## Date

2026-07-23

## Context

The existing `Page` component provides shared chrome for customer, provider, and general routes, but authenticated order routes bypass it. Adding a mobile menu only to `Page` would leave part of the authenticated application without navigation. The existing menu is static, contains hard-coded and stale destinations, does not filter items by user role, and exposes ten destinations with no registered frontend route. The frontend already uses Angular standalone components, signals, Router, SCSS breakpoints, and Angular CDK.

## Decision

Create one authenticated application shell that owns the shared header, persistent desktop navigation, mobile drawer, footer, and nested route outlet. Nest all authenticated customer, provider, general, and order routes beneath this shell while preserving their existing authorization guards.

Represent navigation with one typed local configuration that distinguishes links from actions and declares role availability. Resolve user-specific paths from the authenticated user rather than embedding a fixed user ID. Include only destinations backed by registered routes; the MVP therefore exposes Personal Data, Addresses, and Logout. Hide cards, payments, security, notifications, preferences, support, professional information, bank account, earnings, and Premium until their routes and product flows are implemented. Both desktop and mobile presentations consume the same resolved configuration.

Implement the mobile drawer as an application component controlled by Angular signals. Use Angular CDK A11y primitives for focus containment and restoration. The drawer closes after navigation, backdrop activation, or Escape. It exposes expanded state, dialog semantics, current-page state, and a non-color active indicator. Opening starts immediately without network access and completes its visual transition within 200 ms.

## Alternatives Considered

### Alternative 1: Extend Page and Wrap Exceptional Routes

- **Description**: Add the adaptive navigation to `Page` and wrap each authenticated route that currently bypasses it.
- **Pros**: Requires a smaller initial route hierarchy change.
- **Cons**: Coverage depends on convention, duplicates layout composition, and allows future authenticated routes to omit navigation.
- **Why rejected**: It cannot enforce the PRD requirement across the authenticated application boundary.

### Alternative 2: Conditionally Render Navigation at the Application Root

- **Description**: Show global navigation from the root component based on authentication and current route.
- **Pros**: Covers top-level order routes without nesting them beneath a new shell.
- **Cons**: Couples public and authenticated layouts to route and authentication transitions and creates SSR and duplicate-chrome risks.
- **Why rejected**: A route-defined authenticated shell provides a clearer boundary and predictable composition.

### Alternative 3: Separate Navigation Configurations by Role

- **Description**: Maintain complete menu lists for each user role.
- **Pros**: Makes role-specific variations explicit.
- **Cons**: Duplicates common destinations and increases cross-device and cross-role drift.
- **Why rejected**: One typed configuration with role metadata is smaller and enforces parity.

## Consequences

### Positive

- Every authenticated route inherits navigation by construction.
- Desktop and mobile cannot drift in labels, grouping, ordering, or availability.
- Navigation path defects and the fixed user ID are corrected in the same delivery.
- Users cannot enter broken flows because unregistered destinations are absent.
- No backend, API, storage, or new package is required.
- Existing Angular CDK capabilities provide accessible focus behavior.

### Negative

- The authenticated route hierarchy changes and requires guard regression coverage.
- Existing customer, provider, and general wrappers must stop duplicating shared chrome.
- Role metadata becomes part of the navigation configuration and must stay aligned with route guards.
- Previously visible but unimplemented destinations disappear until their routes exist.

### Risks

- Route nesting may alter redirects or guard evaluation; mitigate with Cypress coverage for each role and protected flow.
- Incorrect path resolution could expose navigation to an invalid destination; mitigate by exercising every rendered destination in end-to-end tests.
- Drawer focus can escape or fail to restore; mitigate with keyboard-only Cypress scenarios.
- Animation timing can be flaky under load; assert state completion with retryable conditions rather than fixed sleeps.

## Implementation Notes

Preserve current route URLs while changing only their parent layout composition. Reuse existing component directories and the installed Angular CDK; do not add a package or persistence layer. Treat successful router navigation as the close signal for link selection so failed navigation does not silently discard context.
