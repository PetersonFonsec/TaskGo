# ADR-004: Use a global authentication guard with explicit public route metadata

## Status

Accepted

## Date

2026-05-30

## Context

The backend lacked a consistent authentication boundary: token validation existed in an unused guard, public and protected routes were not clearly separated, and downstream decorators decoded JWT payloads directly from request headers.

## Decision

Use a global `AuthGuard` to protect all backend routes by default, and expose explicit `@Public()` metadata for routes that must remain anonymous.

The implementation will:

- apply `AuthGuard` globally through the Nest `APP_GUARD` provider;
- introduce `@Public()` to mark login, registration, forget-password, and public discovery GET endpoints as anonymous;
- reject missing, malformed, invalid, or expired tokens with `UnauthorizedException` before the request reaches controllers;
- store verified token payload on `request[TOKEN_KEY]` for downstream controllers and decorators;
- update the shared `@User()` decorator to return the validated requester identity from `request[TOKEN_KEY]` instead of decoding the raw header.

## Alternatives Considered

### Alternative 1: Route-level guards only

- **Description**: Apply `AuthGuard` manually to each protected controller or handler.
- **Pros**: Avoids global guard complexity.
- **Cons**: High risk of missing protections and inconsistent coverage.
- **Why rejected**: The critical marketplace journey requires a reliable boundary for private journeys.

### Alternative 2: Service-layer authorization only

- **Description**: Validate authentication and ownership inside service methods.
- **Pros**: Authorization logic remains close to domain behavior.
- **Cons**: Requires repeated checks and can leave unguarded endpoints.
- **Why rejected**: The MVP needs a consistent request-level gate before business logic.

## Consequences

### Positive

- A single request boundary enforces authentication consistently across the backend.
- Public discovery routes are clearly marked and preserved.
- Downstream controllers can rely on a verified request identity rather than ad hoc JWT decoding.

### Negative

- Adds a new global guard mechanism and metadata pattern to existing NestJS conventions.
- Requires explicit marking of every public endpoint, which is a new maintenance requirement.

### Risks

- If an endpoint that should be public is not marked with `@Public()`, anonymous users will be blocked.
- If the guard is not registered correctly, the application may reject all requests or bypass protection.

## Implementation Notes

- `AuthGuard` uses `Reflector.getAllAndOverride` to respect `@Public()` metadata.
- `@User()` reads `request[TOKEN_KEY]` and returns the validated token payload.
- Public discovery controllers are updated to mark GET discovery routes as anonymous.
