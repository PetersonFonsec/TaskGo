# ADR-001: Protect the Complete Critical Marketplace Journey

## Status

Accepted

## Date

2026-05-30

## Context

The service audit identified four connected critical risks in the marketplace journey:

- protected actions are reachable without effective authentication;
- order identity and price can be supplied by the client;
- responses expose more personal data than users need;
- address access is not isolated to the authenticated user.

Treating these risks as independent fixes would leave gaps between authentication, resource ownership, transaction integrity, and privacy. The product needs a coherent trust baseline for customer and provider interactions while preserving the public discovery experience.

## Decision

Deliver a complete protection baseline for the critical marketplace journey as the MVP.

The MVP will:

- preserve public access to active categories, services, and public provider profiles;
- require a valid authenticated session for contracting, order management, address management, and private account data;
- allow customers to access only their own orders and addresses;
- allow providers to access and act only on orders associated with their services;
- calculate order prices from the official service value controlled by the platform;
- expose only the minimum contextual personal data required for each user journey;
- block release until critical flows demonstrate zero unauthorized access, no sensitive data exposure in public responses, and no client-side price manipulation.

## Alternatives Considered

### Alternative 1: Authentication Barrier First

- **Description**: Require login for private routes and reduce sensitive response fields first. Add detailed resource ownership rules afterward.
- **Pros**: Faster initial delivery and visible security improvement.
- **Cons**: Leaves residual risk because authenticated users could still access or manipulate resources belonging to other users.
- **Why rejected**: The remaining exposure would affect the same critical customer and provider journeys.

### Alternative 2: Privacy and Transaction Integrity First

- **Description**: Correct price calculation, address isolation, and data minimization first. Expand authentication coverage afterward.
- **Pros**: Improves data handling and order integrity quickly.
- **Cons**: Leaves the structural access-control gap open.
- **Why rejected**: Private workflows must not remain reachable without a valid authenticated session.

## Consequences

### Positive

- Establishes a coherent trust baseline for marketplace transactions.
- Preserves public provider discovery while protecting private workflows.
- Reduces exposure of personal data in line with data-minimization expectations.
- Removes client control over authoritative order pricing.

### Negative

- Requires coordinated changes across several user journeys before the MVP can be released.
- May reveal frontend assumptions that depend on currently overexposed response data.

### Risks

- Existing screens may require adjustment when response payloads become smaller.
- Some currently public workflows may need explicit classification during refinement.

## Implementation Notes

Implementation details will be defined in the TechSpec. This ADR intentionally records the product boundary and release posture only.
