# ADR-003: Derive authoritative order price from service basePrice

## Status

Accepted

## Date

2026-05-30

## Context

The `OrderService.create` method currently accepts an optional `finalPrice` from the client payload and uses it when provided. This allows a client to submit an order with a manipulated authoritative amount, violating the critical requirement that the platform control official order pricing.

## Decision

Calculate the order `finalPrice` exclusively from the current `Service.basePrice` at creation time.

The order creation flow will:

- fetch the selected service record by `serviceId`;
- validate that the service exists and is active;
- derive `finalPrice = service.basePrice` regardless of client-provided price data;
- create the order, payment record, and address snapshot using that authoritative price.

## Alternatives Considered

### Alternative 1: Accept client-provided price with validation

- **Description**: Compare client price against service price and reject mismatches.
- **Pros**: Preserves client payload shape; allows tolerance checks.
- **Cons**: Still relies on client data and adds complexity.
- **Why rejected**: The requirement is to prevent client-defined authoritative pricing entirely.

### Alternative 2: Use a dedicated pricing service

- **Description**: Introduce a pricing service that computes finalPrice from basePrice, fees, and promotions.
- **Pros**: Supports future pricing rules and discounts.
- **Cons**: Adds abstraction beyond the MVP scope.
- **Why rejected**: The MVP only needs current official service price and must avoid unnecessary architecture changes.

## Consequences

### Positive

- Eliminates client influence over authoritative order price.
- Keeps pricing logic simple and predictable in the marketplace MVP.
- Aligns order creation with the PRD goal of official pricing integrity.

### Negative

- Future pricing rules or promotions will require a controlled extension point.
- The frontend must stop sending client `finalPrice` values or ignore them safely.

### Risks

- Existing consumers may rely on `finalPrice` in the request body.
- If `Service.basePrice` is missing or stale for a service, order creation will fail.

## Implementation Notes

- Remove or ignore `finalPrice` from `CreateOrderDto` in protected order creation routes.
- In `OrderService.create`, always use `service.basePrice` for order and payment amount.
- Add a migration or validation rule to ensure active services have a non-null `basePrice`.
