# ADR-002: Enforce ownership and privacy via guard-level authorization

## Status

Accepted

## Date

2026-05-30

## Context

The backend currently exposes order and address endpoints without a global authorization boundary. The `AuthGuard` exists, but it is not applied consistently, and several private resource queries accept client-supplied identifiers without binding them to the authenticated user.

## Decision

Implement ownership and privacy enforcement primarily through guard-level authorization.

The system will:

- apply a global or route metadata-driven authorization guard to private routes;
- extract authenticated user identity and role from the JWT once per request;
- use route metadata to declare required ownership semantics for customer and provider journeys;
- block requests that attempt to access or mutate resources outside authenticated identity before the service layer executes business logic.

## Alternatives Considered

### Alternative 1: Service-layer checks only

- **Description**: Keep guards simple and enforce ownership inside each service method.
- **Pros**: Keeps authorization logic with data access code; service methods can inspect full resource state.
- **Cons**: Duplicates ownership checks across services; increases risk of a missing check causing unauthorized access.
- **Why rejected**: The critical marketplace journey requires a centralized enforcement boundary to avoid inconsistent protection.

### Alternative 2: Dual guard and service checks

- **Description**: Implement both route guards and service-level ownership checks for every protected endpoint.
- **Pros**: Provides defense in depth.
- **Cons**: Adds extra complexity and duplication.
- **Why rejected**: The MVP can meet the requirement with primary guard-level enforcement and targeted service-side validation for mutating actions.

## Consequences

### Positive

- Authorization is enforced consistently at the request boundary.
- Private routes can be classified explicitly as customer-only or provider-only.
- The request pipeline can fail fast before expensive queries or data shaping.

### Negative

- Some service methods may still need minimal ownership validation for safety, increasing implementation effort slightly.
- Guard metadata must be kept in sync with route definitions.

### Risks

- Incomplete guard coverage would leave critical journeys unprotected.
- A guard relying only on `user.id` requires accurate role resolution for provider actions.
- Global guard changes must be tested carefully to avoid breaking public discovery routes.

## Implementation Notes

- Use `AuthGuard` with `@UseGuards` or a global guard provider for private routes.
- Add a request-scoped authenticated identity object to `req[TOKEN_KEY]`.
- Introduce a metadata helper such as `@OwnerScope('customer')` or `@OwnerScope('provider')` for route classification.
- Private route controllers should require the guard and use the authenticated ID rather than client-provided IDs where possible.
