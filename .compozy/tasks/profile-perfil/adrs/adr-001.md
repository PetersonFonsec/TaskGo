# ADR-001: Modular phased rollout for Profile screens

## Status

Accepted

## Date

2026-05-30

## Context

We need editable profile screens where both clients and professionals can update contact information, addresses, and related personal details. The product should support an MVP launch while minimizing scope, user confusion, and development overhead.

## Decision

Adopt a modular, phased rollout: deliver a shared core profile (contact info, primary address, display name, avatar) for both user roles in the MVP, then add role-specific sections (professional credentials, service areas, business hours, client-specific preferences) in subsequent phases.

## Alternatives Considered

### Alternative 1: Unified adaptive form

- **Description**: One edit screen that dynamically shows/hides fields based on role.
- **Pros**: Single UI to maintain; consistent experience across roles; faster for parity.
- **Cons**: Can become cluttered; harder to optimize for role-specific needs.
- **Why rejected**: Prefer progressive delivery and clearer launch scope.

### Alternative 2: Role-specific flows

- **Description**: Separate edit screens and flows for clients and professionals, each tailored to role needs.
- **Pros**: Fully tailored UX; easier to optimize fields for each role.
- **Cons**: More upfront design and QA work; duplicated UI components; slower to reach parity.
- **Why rejected**: Increased initial effort and slower MVP delivery.

## Consequences

### Positive

- Rapid MVP delivery with clear scope.
- Shared components reduce duplication.
- Roadmap-friendly: role-specific features can be iterated independently.

### Negative

- Some role-specific needs will be deferred beyond MVP.
- Possible minor refactors when adding role-specific sections.

### Risks

- Users may request missing role-specific fields immediately; mitigate by clearly communicating phased features and adding critical fields to MVP if needed.

## Implementation Notes

- Design and implement a clear schema for `core` vs `role` profile fields to ease future additions.
- Ensure edit/save UX is consistent across roles and that validation is role-aware server-side.
